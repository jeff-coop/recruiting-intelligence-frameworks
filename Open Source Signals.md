# Open Source Signals

This page documents how I think about open source when evaluating technical talent.

Open source is not required to be a strong engineer.

Many exceptional engineers work on proprietary systems and may have little public activity.

However, when open source experience exists, it can provide useful insight into technical interests, learning habits, collaboration, communication, and long-term commitment.

The goal is not to rank candidates.

The goal is to better understand how engineers engage with technical communities and contribute beyond their day-to-day work.

---

# Why Open Source Matters

Open source creates opportunities to:

* Learn publicly
* Collaborate globally
* Build technical credibility
* Contribute to real-world software
* Develop communication skills

Open source often provides signals that are difficult to observe through resumes alone.

---

# Types of Contributions

Not all contributions involve writing code.

Examples:

* Documentation
* Bug Reports
* Feature Requests
* Community Support
* Testing
* Code Contributions

All can provide useful signals.

---

# Contributor vs Maintainer

## Contributor

Examples:

* Pull Requests
* Bug Fixes
* Documentation Updates

Signals:

* Collaboration
* Initiative
* Learning

---

## Maintainer

Examples:

* Managing Issues
* Reviewing Pull Requests
* Setting Project Direction
* Supporting Community Members

Signals:

* Leadership
* Ownership
* Communication
* Technical Depth

Maintainer experience is often one of the strongest open-source signals.

---

# Long-Term Commitment

One contribution is useful.

Years of involvement can be even more interesting.

Questions:

* How long have they been involved?
* Do they return consistently?
* Have responsibilities increased over time?

Long-term participation often demonstrates commitment and growth.

---

# Technical Communities

Open source often exposes engineers to communities that influence how they learn and develop.

Examples:

* Kubernetes
* Linux
* PostgreSQL
* Apache
* PyTorch
* OpenTelemetry

The communities people participate in can reveal technical interests and career direction.

# Communication Signals

One of the most overlooked aspects of open source.

Examples:

* Documentation
* Design Discussions
* Community Support
* Code Reviews

Strong communication frequently appears alongside strong technical ability.

---

# Signals I Find Most Interesting

Examples:

* Maintainer Experience
* Long-Term Contributions
* Technical Writing
* Community Leadership
* Infrastructure Projects
* Developer Tooling
* Research-Oriented Projects

---

# Signals I Care Less About

Examples:

* Number of Repositories
* Number of Followers
* Number of Stars Alone

Without context, these metrics are often weak indicators.

---

# Questions I'm Trying To Answer

When reviewing open source activity, I am usually asking:

* What interests this person?
* What communities are they involved in?
* How do they collaborate?
* What motivates them?
* How have they grown over time?

The goal is not to find perfect contributors.

The goal is to better understand the person behind the contributions.
