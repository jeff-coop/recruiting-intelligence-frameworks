# GitHub Signals

This page documents how I think about GitHub when evaluating technical talent.

GitHub should never be used as a pass/fail hiring signal.

Many exceptional engineers have little or no public GitHub activity.

However, when available, GitHub can provide useful insight into technical interests, learning habits, ownership, collaboration, and community involvement.

The goal is not to evaluate code quality.

The goal is to better understand how engineers spend time outside of their day-to-day responsibilities.

---

# First Thing I Look At

## Pinned Repositories

Pinned repositories are often the fastest way to understand what a candidate wants others to see.

Questions:

* What projects are highlighted?
* What technologies appear repeatedly?
* Are these personal projects?
* Open source contributions?
* Research projects?
* Developer tools?

Pinned repositories often reveal interests more clearly than resumes.

---

# Repository Quality

I care far more about quality than quantity.

Things I look for:

* Clear README
* Documentation
* Project Organization
* Thoughtful Explanations
* Long-Term Maintenance

A single strong project is often more interesting than dozens of unfinished repositories.

---

# Contribution Activity

Contribution graphs can provide useful context.

Questions:

* Is activity consistent?
* Are projects maintained over time?
* Are contributions concentrated around specific interests?

Consistency is usually more meaningful than volume.

---

# Open Source Contributions

One of the strongest signals when present.

Examples:

* Pull Requests
* Bug Fixes
* Feature Contributions
* Documentation Improvements
* Community Support

Open source contributions often demonstrate:

* Curiosity
* Collaboration
* Communication
* Technical Depth

---

# Pull Requests

Pull requests often provide insight beyond code.

Things I pay attention to:

* Technical Discussions
* Collaboration
* Review Participation
* Problem Solving
* Communication Style

Strong pull requests often demonstrate how engineers work with others.

---

# Stars

Stars are usually weak signals by themselves.

However, they can reveal interests.

Questions:

* Which projects do they follow?
* Which communities are they interested in?
* What technologies appear repeatedly?

Examples:

* Kubernetes
* PostgreSQL
* Kafka
* PyTorch
* ClickHouse

Stars often reveal learning interests before they appear on resumes.

---

# Forks

Forks require context.

A fork alone means very little.

More useful questions:

* Was the repository modified?
* Were contributions made?
* Did learning occur?
* Was a project built from it?

---

# Technical Writing

One of the most underrated signals.

Examples:

* README Files
* Documentation
* Technical Blogs
* Tutorials
* Project Explanations

Strong engineers often communicate clearly.

---

# Breadth vs Depth

## Breadth

Many projects across different technologies.

Potential Signals:

* Curiosity
* Exploration
* Learning

---

## Depth

Long-term commitment to a project or ecosystem.

Potential Signals:

* Ownership
* Technical Depth
* Persistence

Both can be valuable depending on the individual.

---

# Personal Projects

Projects can provide useful insight into:

* Motivation
* Learning Habits
* Technical Interests
* Problem Solving

Questions:

* What did they choose to build?
* Why did they build it?
* How long did they invest in it?

---

# Signals I Find Most Interesting

Examples:

* Long-Term Projects
* Open Source Contributions
* Maintainer Experience
* Technical Writing
* Research Projects
* Developer Tools
* Infrastructure Projects
* AI Projects
* Community Involvement

---

# Signals I Care Less About

Examples:

* Raw Commit Counts
* Number of Followers
* Number of Repositories
* Number of Stars Received

These metrics often lack context and can be misleading.

---

# Questions I'm Trying To Answer

When reviewing a GitHub profile, I am usually asking:

* What interests this person?
* What do they learn outside of work?
* What do they build?
* How do they collaborate?
* How do they communicate?
* What communities are they involved in?

The goal is not to judge.

The goal is to understand.
